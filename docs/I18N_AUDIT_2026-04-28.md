# Auditoría i18n — Strings hardcoded en español

**Autor:** David Cabezas · **Fecha:** 2026-04-28
**Scope:** todo `frontend/src/**` excluyendo `i18n/*.js` (que son archivos de traducción, OK).
**Norma:** strings user-facing deben pasar por `t('key')` y vivir en `frontend/src/i18n/{en,es,ar}.js`.

**Total estimado:** ~170 strings hardcoded. Esfuerzo cobertura completa: **24-32h** (4 tiers).

---

## Convenciones

- ❌ Hardcoded en JSX/JS = bug i18n
- ✅ Strings dentro de `i18n/*.js` = OK (son las traducciones)
- 🔵 Comentarios en español = OK (no se renderizan al usuario)
- 🔵 Variables/keys que coinciden con backend (ej. `'CREADO'`, `'PROGRAMADO'`) = OK (son enums)

## Cómo NO repetir el problema

1. **Política**: ningún string visible al usuario va hardcoded. Si el feature es nuevo, agregar key en `en.js` + `es.js` + `ar.js` desde el primer commit.
2. **Pre-commit hook** (a crear): linter custom que detecte strings con palabras españolas comunes en JSX no envueltas en `t()`.
3. **Code review**: cualquier PR que agregue UI debe pasar por el ojo de "¿esto se ve en EN?".
4. **Linter ESLint regla**: existe `react-i18n/no-untranslated` o similar — evaluar si es viable activarlo.

---

## Notas para el migrante

- **Plurales**: usar `i18next` interpolation (`{{count}} acciones sincronizadas`) en lugar de strings hardcoded con plurales.
- **Fechas**: usar `lang === 'es' ? 'es-CL' : ...` para `toLocaleDateString` — varios componentes ya lo hacen bien.
- **Comments en español**: NO los toques — son comentarios internos, no rompen i18n.
- **Backend strings**: los errores que vienen del backend (ej. `detail` de FastAPI) están en español y se muestran tal cual. Eventualmente migrar también el backend a i18n via header `Accept-Language`.
- **Comodín**: si una clave de traducción se usa en 1 solo lugar, puede vivir junto al componente como `componentName.field`. Si se reusa, ponerla bajo `common.field`.
