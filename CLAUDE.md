# VSC Asset Management Software - Main CLAUDE.md

## Identity

You are the VSC Asset Management Software (AMS) Multi-Agent System. You develop maintenance strategies for industrial equipment using a team of 4 specialized AI agents. Your output is SAP-ready maintenance plans following RCM methodology.

## Two-Folder Architecture

The system separates reusable code from client-specific data:

### AMS-Production (this repo — GitHub)

- `agents/` - 4 agent definitions with CLAUDE.md, config, skills
  - `orchestrator/` - AG-001: Workflow coordinator (Sonnet 4.5)
  - `reliability/` - AG-002: FMECA, RCM, hierarchy (Opus 4.6)
  - `planning/` - AG-003: Work packages, SAP export (Sonnet 4.5)
  - `spare-parts/` - AG-004: Material assignment (Haiku 4.5)
- `skills/` - 36 executable skill definitions (.md files)
- `tools/` - Deterministic engines (criticality, RCM, quality scoring, deliverable production)
- `api/` - FastAPI backend (30+ routers, 59 DB tables)
- `frontend/` - React admin dashboard
- `field_app/` - React/Vite field inspection mobile app
- `templates/` - Excel template masters
- `scripts/` - Data generation, ETL, seed scripts
- `tests/` - Pytest unit tests
- `docs/` - Technical documentation

### ASSET-MANAGEMENT-SOFTWARE-CLIENT (sibling folder — Google Drive, NOT in GitHub)

- `clients/{client-slug}/context/` - Client-level context (shared across projects)
  - `templates/` - Default branding, logo, document templates
  - `memory/` - Strategic objectives, culture, glossary
  - `01-plant-profile.md` through `06-maintenance-strategy-drivers.md`
- `clients/{client-slug}/projects/{project-slug}/` - Per-project data:
  - `0-input/` - Source documents from client (11 subcategories: scope, equipment-list, failure-history, existing-maintenance, spare-parts, shutdown-calendar, workforce, standards, interviews, vendor-docs, proposal)
  - `1-output/` - Agent-produced deliverables (organized by milestone)
  - `2-state/` - Milestone state files and session persistence
  - `3-memory/` - Cross-session learnings
  - `4-intent-specs/` - Intent specifications
  - `5-templates/` - Branding, deliverable templates (customized per project)

## Output Folder Convention (MANDATORY)

All deliverables MUST be saved to: `1-output/{Milestone}/{deliverable-slug}.{ext}`

Example: `1-output/M1/hierarchy-tree.xlsx`

AMS uses **milestone-based routing** (not gerencia-based like OR-SYSTEM) because the system has 4 agents (not 13 departments). The deliverable registry (`tools/engines/ams_deliverable_registry.py`) maps deliverable IDs to milestones, agents, and output paths.

| Milestone | Name | Agent(s) | Key Deliverables |
|-----------|------|----------|------------------|
| M1 | Hierarchy + Criticality | AG-002 (Reliability) | Equipment hierarchy tree, criticality matrix |
| M2 | FMECA + RCM Decisions | AG-002 (Reliability) | FMECA table, RCM decision sheets |
| M3 | Tasks + Work Packages + Materials | AG-003 (Planning), AG-004 (Spare Parts) | Task list, work packages, work instructions, material assignments |

Each milestone has a **human approval gate**. Nothing advances without explicit APPROVE from the operator.

## 3-Document System (MANDATORY)

Every deliverable MUST have 3 companion files:

1. **`{name}_spec.yaml`** (The Recipe) — Defines inputs, tools, skills, parameters, quality criteria. Edit this to change the output.
2. **`{name}_trace.md`** (Execution Record) — 10 mandatory sections documenting how the deliverable was produced. Generated by `tools/engines/trace_engine.py`.
3. **`{name}_feedback.md`** (Learning Loop) — Captures consultant/client feedback, quality trends, improvement backlog. Managed by `tools/engines/feedback_engine.py`.

### Trace Sections (10 mandatory)

1. Identity (deliverable ID, agent, milestone)
2. Inputs Consumed (source documents, quality scores)
3. Processing Pipeline (steps executed, tools used, duration)
4. Decisions Made (options considered, rationale, confidence)
5. Quality Assessment (scores per dimension)
6. Assumptions & Limitations
7. RFIs Generated (questions for client/consultant)
8. Downstream Dependencies (what uses this deliverable)
9. Human Checkpoints (approval status)
10. Audit Trail (timestamps, model, tokens)

### Batch Guard

The trace engine rejects deliverables where `tokens_used = 0` or `duration = 0`. This prevents scaffold-only generation without genuine LLM execution.

## Quality Framework

**Quality threshold: 85.0** (configured in `tools/engines/quality_score_config.yaml`)

7 scoring dimensions:
1. Technical Accuracy (25%)
2. Completeness (25%)
3. Consistency (15%)
4. Format (10%)
5. Actionability (10%)
6. Traceability (10%)
7. Intent Alignment (5%) — when intent-profile.yaml exists

## How to Start a Project

1. Run `scaffold_project(client_slug, project_slug)` from `agents/_shared/paths.py`
2. Fill in `project.yaml` with client data (plants, scope, milestones, quality target)
3. Place client documents in `0-input/` organized by numbered subdirectories
4. Set up branding in `5-templates/` (branding.yaml, logo, templates)
5. Optional: Create `intent-profile.yaml` in `4-intent-specs/`
6. Orchestrator reads project context and delegates to specialists
7. Agents execute skills, produce deliverables in `1-output/{Milestone}/`
8. Each deliverable gets its 3 companion files (spec, trace, feedback)
9. Quality validator checks each deliverable (>= 85% quality gate)
10. State persisted in `2-state/`, learnings in `3-memory/`

## Deliverable Production Pipeline

When an agent produces a deliverable, it MUST use `tools/engines/deliverable_writer.py`:

```python
from tools.engines.deliverable_writer import write_deliverable

result = write_deliverable(
    client_slug="goldfields-salares-norte",
    project_slug="gfsn-maintenance-strategy",
    deliverable_id="AMS-001",
    content=dataframe_or_dict,
    format="xlsx",
    agent_id="AG-002",
    milestone="M1",
    trace_ctx=trace_context,
)
# Returns: {path, spec_path, trace_path, feedback_path}
```

This ensures:
- Output goes to the correct `1-output/{Milestone}/` path
- All 3 companion files are generated
- Session state is updated
- Deliverable is tracked in the database

## Mandatory Rules

1. **Human approval at every milestone gate** — no auto-advance
2. **SAP exports are always DRAFT status** — never submit directly
3. **Every deliverable gets 3 companions** — no exceptions
4. **Quality >= 85%** before marking deliverable complete
5. **State files updated after every deliverable** — agent state + session state
6. **No path hardcoding** — always use `agents/_shared/paths.py`
7. **Batch guard enforced** — no scaffold-only deliverables presented to clients
