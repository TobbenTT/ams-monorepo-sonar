**Source:** `or-strategy-template-example-01--MAM-PRJ00-TD-GDL-00042-Operational-Readiness-Strategy-Template.docx`

| Section | Content | Pages |
|---------|---------|-------|
| 1. Purpose & Scope | Why OR is needed, program boundaries | 1 |
| 2. Project Context | Project description, key parameters, complexity | 2 |
| 3. OR Vision & Objectives | What OR will achieve, measurable goals | 1 |

| 6. Stage-Gate Framework | Gates G0-G5 with criteria and schedule | 3 |

- Project name, client, location

## 3. Facility Handover Notice Template

**Source:** `ASCS-514165-OR-FHN-51001-Template-Facility-Handover-Notice.docx`

### TCCC Model (Transferred, Commissioned, Certified, Closed)
| Stage | Description | Documentation |
|-------|-------------|---------------|
| Transferred | System physically complete, handed from construction to commissioning | Mechanical completion certificate |
| Commissioned | System tested and verified to meet design specifications | Commissioning test records |
| Certified | System certified safe to operate by competent authority | Safety certification |
| Closed | All punch list items resolved, system in normal operations | Closeout certificate |

### Facility Handover Notice (FHN) Content
1. System identification and boundaries
2. Status of mechanical completion
3. Status of commissioning activities
4. Punch list summary (A/B/C items)
5. Outstanding work and responsible parties
6. Safety and environmental clearances
7. Operating procedure readiness
8. Maintenance readiness verification
9. Trained operators confirmation
10. Handover acceptance signatures

## 6. Template Usage Guidelines

### When to Use Each Template

| Situation | Template | Customization Level |
|-----------|----------|-------------------|
| New OR program starting | OR Strategy Template | High (project-specific) |
| Moving from strategy to planning | OR Plan Template | Very High (all sections) |
| System handover during commissioning | Facility Handover Notice | Moderate (system-specific) |
| Workstream-level planning | OR Business Plans examples | High (adapt to project) |
| Pre-feasibility stage | Pre-feasibility example | Moderate (simplified version) |

### Customization Rules
1. **Always customize**: Project name, client, location, dates, team
2. **Adapt scope**: Add/remove workstream sections based on project complexity
3. **Scale depth**: Smaller projects need shorter documents
4. **Follow client standards**: If client has document templates, adapt to their format
5. **Maintain consistency**: All documents in a project use the same terminology, formatting, and cross-references
